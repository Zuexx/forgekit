# Design: Comprehensive API Documentation

## Overview

This design implements professional-grade API documentation using XML documentation comments that:
- Display in IDE IntelliSense during development
- Generate into OpenAPI specification for integration with Scalar UI
- Document business intent, constraints, and error scenarios
- Support future API documentation generation and client SDK creation

## Documentation Layers

### 1. Source Code Documentation (XML Comments)

All public API surfaces carry XML documentation comments:
- Display in IDE tooltips when hovering or using IntelliSense
- Include in OpenAPI specification (if configured)
- Support Scalar UI with descriptions and metadata
- Provide reference documentation for developers

**Locations:**
```
Api/Features/*/Commands/      → Command classes and handlers
Api/Features/*/Queries/       → Query classes and handlers
Api/Models/                   → Request/Response DTOs
Api/Exceptions/               → Exception types
Api/Services/                 → Application services
Api/Domain/Services/          → Domain services
```

### 2. Documentation Propagation

XML documentation flows:
```
Source Code (XML comments)
    ↓
Api.xml (generated by build)
    ↓
OpenAPI Schema (if configured)
    ↓
Scalar UI + IDE IntelliSense
```

## Documentation Structure

### Handler Documentation Pattern
```csharp
/// <summary>
/// Creates a new entity resource with specified properties.
/// </summary>
/// <remarks>
/// This command orchestrates the following:
/// - Validates input parameters
/// - Checks authorization and business rules
/// - Creates the entity aggregate
/// - Persists to database with audit context
/// - Logs the operation for observability
/// 
/// Business Rules:
/// - Entity must satisfy business constraints
/// - User must have required permissions
/// - Related entities must exist
/// 
/// Error Handling:
/// - NotFoundException: When referenced entity not found
/// - UnauthorizedException: When user lacks permission
/// - ValidationException: When validation fails
/// </remarks>
public class CreateEntityCommand : ICommand<CreateEntityResponse>
{
    /// <summary>Gets the unique identifier of the resource.</summary>
    public required string Id { get; set; }

    /// <summary>Gets the entity name or description.</summary>
    public required string Name { get; set; }
}

/// <summary>Handles entity creation.</summary>
public class CreateEntityHandler : IRequestHandler<CreateEntityCommand, CreateEntityResponse>
{

    /// <exception cref="NotFoundException">When referenced resource not found</exception>
    /// <exception cref="UnauthorizedException">When authorization fails</exception>
    /// <exception cref="ValidationException">When validation fails</exception>
    public async Task<CreateEntityResponse> Handle(
        CreateEntityCommand request,
        CancellationToken cancellationToken)
    {
        // Implementation...
    }
}
```

### Response Model Documentation Pattern
```csharp
/// <summary>
/// Response model for entity creation.
/// </summary>
/// <remarks>
/// Contains the newly created entity details and audit information.
/// </remarks>
public class CreateEntityResponse
{
    /// <summary>The system-generated unique identifier.</summary>
    public required string Id { get; set; }

    /// <summary>The entity name or description.</summary>
    public required string Name { get; set; }

    /// <summary>Creation timestamp (UTC, ISO 8601).</summary>
    public required DateTime CreatedAt { get; set; }

    /// <summary>User who created this entity (audit trail).</summary>
    public required string CreatedBy { get; set; }
}
```

### Error Response Documentation Pattern
```csharp
/// <summary>
/// Standard error response format.
/// </summary>
/// <remarks>
/// All API errors use this consistent format for unified error handling.
/// </remarks>
public class ErrorResponse
{
    /// <summary>Human-readable error message.</summary>
    public required string Message { get; set; }

    /// <summary>Machine-readable error code.</summary>
    /// <remarks>Examples: VALIDATION_ERROR, NOT_FOUND, UNAUTHORIZED, CONFLICT</remarks>
    public string? Code { get; set; }

    /// <summary>Timestamp when error occurred (UTC, ISO 8601).</summary>
    public required DateTime Timestamp { get; set; }

    /// <summary>Field-level validation errors.</summary>
    /// <remarks>
    /// Only for validation errors. Dictionary key is field name,
    /// value is array of error messages for that field.
    /// </remarks>
    public Dictionary<string, string[]>? Errors { get; set; }

    /// <summary>Request trace ID for server log correlation.</summary>
    public string? TraceId { get; set; }
}
```

## Documentation Standards

### Summary Guidelines
- Keep to one sentence (< 80 characters)
- Describe what the class/method does, not how
- For methods: Start with verb ("Creates", "Updates", "Retrieves", "Validates")
- For classes: Start with noun phrase ("Request for...", "Response containing...")
- Use present tense
- Avoid redundancy (don't repeat class name)

### Remarks Guidelines
- Add business context and workflow
- Explain state transitions
- Document business rules and constraints
- Specify validation rules
- List error conditions and exceptions
- Explain side effects (audit, logging, notifications)
- Keep technical implementation details minimal
- Use examples when helpful

### Exception Documentation Guidelines
```csharp
/// <exception cref="SpecificException">
/// Brief description of when this exception is thrown.
/// Additional detail on conditions that cause this exception.
/// HTTP status code and error code returned.
/// </exception>
```

### Parameter Documentation Guidelines
- Document constraints (required, format, range)
- Document valid values (enum, allowed ranges)
- Document format (email format, date format, ID format)
- Reference related parameters
- Mention side effects of parameter values

### Return Value Documentation Guidelines
- Describe what the response contains
- Mention when response fields are populated vs empty
- Explain response format and structure
- Note any transformations applied

## API.xml Generation

### Configuration

Update `Api.csproj`:
```xml
<PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <!-- Enable XML documentation generation -->
    <GenerateDocumentationFile>true</GenerateDocumentationFile>
    <DocumentationFile>bin\$(Configuration)\$(TargetFramework)\Api.xml</DocumentationFile>
    <!-- Suppress missing XML comment warnings for internal members -->
    <NoWarn>1591</NoWarn>
</PropertyGroup>
```

### Build Process
1. Build project: `dotnet build`
2. Generate Api.xml: Automatically created in output directory
3. Output location: `bin\Debug\net10.0\Api.xml`

### XML Structure
```xml
<?xml version="1.0"?>
<doc>
  <assembly>
    <name>Api</name>
  </assembly>
  <members>
    <member name="T:Api.Features.Commands.CreateEntityCommand">
      <summary>Creates a new entity resource...</summary>
      <remarks>Business logic description...</remarks>
    </member>
    <member name="P:Api.Features.Commands.CreateEntityCommand.Id">
      <summary>The unique identifier...</summary>
    </member>
  </members>
</doc>
```

## Integration with Scalar UI

When configured, Scalar automatically:
1. Reads OpenAPI specification from `/openapi/v1.json`
2. Looks up XML documentation in Api.xml
3. Displays summaries in API operation descriptions
4. Shows remarks in expanded documentation
5. Displays parameter descriptions
6. Shows exception/error information

## Documentation Maintenance

For each new feature:
1. Write XML documentation comments as you write code
2. Include business context in remarks
3. Document all exceptions that can be thrown
4. Keep documentation current with code changes
5. Review documentation in code reviews
6. Test Scalar UI displays documentation correctly
7. Verify IDE tooltips show expected content

## Documentation Quality Checklist

For each public API:
- ✅ Summary tag (one sentence)
- ✅ Remarks tag (if more context needed)
- ✅ Param tags (for all parameters)
- ✅ Returns tag (for return value)
- ✅ Exception tags (for all thrown exceptions)
- ✅ Clear business intent expressed
- ✅ Constraints and rules documented
- ✅ Error scenarios documented
